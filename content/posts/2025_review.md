# 2025 年终总结

今年总体来说有很多全新的体验。

## 找实习

今年已经到了投暑期实习的时候了，我第一次知道暑期实习可以开的这么早，字节跳动的暑期实习 ByteIntern 专项是在年后开工的那段时间就开了。我在那时候简历还没有打磨好，算法也没怎么刷，但看到有人说就算你投递了也很难马上有面试，因为本身投的人就多，加上学校水平不行，HR 也很难在短期内通过。事实证明可能是正确的，因为我直到四月初才收到字节的一面。

我总共有几次面试呢，其实不多，字节是我十分后悔的一个，因为我总感觉是不是当初因为当初答错了一个点导致没通过一面。不过找工作其实也就是这样，总是看运气的。不止是找工作，很多事情都要看运气。我在以前实际上对很喜欢证明论和数理逻辑那种确定的感觉。我很喜欢计算机，我喜欢程序，大概就是因为这太确定了，至少相比于现实世界来说确实如此。虽然我这么说确定性，但我也认同，完全的随机是公平的体现，为了这个公平当然要引入随机数的出现，不过这和我刚刚说的其实不在一个层面上（

我忘记不了第一次面试，是腾讯的 PCG 事业群，那次我很狼狈，因为准备的不好导致挂掉了。这个面试我并不后悔，因为后悔的前提是感觉能成功，但关于那次面试，我差的一星半点，那时候我真是没准备好就收到了面试邀约。但是有这次面试的经历也很好，因为反正总是要有第一次面试，早点面试虽然过的概率不大，但是总是有了个头，总会有好的结尾的，对吧？

还有什么面试值得吐槽吗，还有腾讯云智，这个好像是挂在腾讯云下面的公司。西安腾讯云智那次面试让我感到很疑惑，因为我从来没有在它岗位描述中看到任何什么和大数据强相关的东西，我投的不是后端吗，为什么面试官一副你就应该对大数据有学习的样子，原来现在互联网后端还有大数据的要求了吗。

之后就是小米，最后我拿到了小米的暑期实习，做了一开始我就很想做的 —— 操作系统开发，虽然不是最开始想的 Linux kernel 研发（

突然想起来我还找中科院 PLCT 实验室的远程实习来着，但也失败了。现在回想起来我也是在一个错误的时间投递，我当时幼稚的以为没什么编译器开发实验就可以投递编译器开发的实习岗位了。

## 刷课

好像没有刷课了，因为今年上半年我基本都在准备找实习，下半年找到了实习，基本都是做工作相关的内容了。但我有一个刷课的计划，那就是韩国科学技术院的 CS420，这是一个编译原理的课程，课程的 lab 是用 Rust 写一个 C 的编译器，cs diy 这样评价这门课程

> 这门课程提供了一个 Rust 语言编写的编译器 KECC(KAIST Educational C Compiler) 框架代码，在此基础上构建编译器的一些核心部分，包括语法树打印、SSA 中间代码生成、中间代码优化（包括 CFG 简化，GVN 等）以及目标代码生成。与其他编译器课程相比，本课程主要有以下特点：
>
> - 该课程面向真实的 C 语言而非自定义的玩具语言，并且会使用 C 语言 Fuzzing 工具 (Csmith) 进行测试。
> - 该课程注重实用性，对编译器前端理论涉及很少，第一个实验也是直接从抽象语法树的遍历开始，并不要求你手动完成前端。课程重点关注于 IR 的设计、生成和优化，以及 RISC-V 汇编代码生成。本课程对于理解和学习 LLVM 也很有帮助。
> - 课程配套视频有详细的代码讲解，对初学者友好。

看起来很有意思，我今年上半年有尝试过刷北大的编译原理实践那个 lab，这门课的 lab  是用 C/C++/Rust 任选一个写一个编译器的编译部分，也就从源代码到 IR 再到汇编语言这个标准流程。我没写完这个课程的 lab 就放弃了，转头去准备面试，所以我在年底，也就是现在开始学习 Rust，然后刷这个 CS420

还有一个是巴西的米纳斯吉拉斯联邦大学的一个课程，编号是 DCC888，这门课专注于编译器中端的优化部分。这门课程是偶然间在 Reddit 上看到的，好像很可以？让我学完 CS420 后再看看这门课吧。

## 关于实习

我对 git 的熟悉使用都是这个实习带来的，因为我以前其实也不用很多 git 操作，也就是 add/commit/pull/push，其他的有了解但用的不多。

在实习中我相当多的使用 checkout/switch、rebase、stash、diff，此时我不得不说 git 真的是一个好东西啊，分布式的源代码项目版本管理系统，真的很好用。

同时我开始使用起了 tmux，在实习之前我从来不用 tmux。我要么用终端模拟器自带的标签栏功能，要么就多开 workspace 然后让终端模拟器铺满整个屏幕，但现在我意识到 tmux 还是有好处的，因为以前我能忍受调试目标和调试器不在一个屏幕里，现在我忍不了了。不过我感觉有一个对 tmux 批评很有意思，那就是图形渲染也要 tmux 来做这件事对 tmux 来说有些重了，终端复用器的功能大可以交给终端本身，Ghostty 还是 Kitty 来着就好像是这个思路？但我还没尝试。

## 未来展望 -- 关于 2026

首先，我需要把 Rust 学完，然后刷完 CS420，我希望借此我可以对编译器有一个认识，然后就可以开始学习现代化编译器了。如果有时间的话，可以再看看 DCC888 关于这个程序分析和优化的课程。

为什么我会想这样，因为我想成为 LLVM 开发者。因为一些原因，我看了一下 LLVM 中 compiler-rt 的代码，所以我想再学习一下现代化编译器的架构理念，借此成为 LLVM 开发者。虽然 compiler-rt 只是 LLVM 的运行时库，和编译的过程不怎么搭边（

我想学习 Zig，然后用 Zig 写一个操作系统内核，我之前写的 OS kernel 设计架构太老了，我希望能我这次能用一些现代化前沿的设计理念。

我想学习 Zig 的原因是，它让我感觉它是一个现代化的 C，可能会有人说 Why not Rust，但是 Rust 写的 OS kernel 已经遍地都是了，我写起来太没意思了。

不过用 Zig 写 OS kernel 的优先级不会太高，比起 Rust 来说，Zig 还是太小众了一些。

我要先学习 Rust，因为我一方面可以学习编译器，另一方面是我感觉以后系统开发可能会有它的一席之地。当然学习编译器不是学习 Rust 才能学，甚至 Clang/LLVM 还是 C++ 写的，那为什么不写那些 C/C++ 写 lab 的课程然后直接到 LLVM 的 good first issue 呢，当然是刚刚提到的 "另一方面" 在作祟。

我希望能培养出摄影的爱好，让我好好记录下我看到的事物吧😋
